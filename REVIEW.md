> Контекст: проектът е Expo SDK 54 + TypeScript + Expo Router + NativeWind + expo-sqlite/Drizzle + Zustand.

Този документ е подреден като задачи за изпълнение. Приоритет **P1** = да се направи преди Phase 4; **P2** = важно, но не блокиращо; **P3** = подготовка специално за Phase 4.

## Обща оценка

Сериозна, чиста кодова база, която следва ROADMAP. Строги типове, derived логиката е изнесена в `lib/domain.ts`, централизирана тема, преизползваеми компоненти. Готово за реална лична употреба. Забележките по-долу са основно за неща, които ще създадат проблем при добавянето на облак/синхронизация, ако не се адресират сега, докато данните са малко и за изхвърляне.

### Какво е добре (да не се чупи при рефакторинг)
- Енумни типове в `schema.ts`; чисти `$inferSelect/$inferInsert` типове.
- Derived логика в `lib/domain.ts` (`isPaymentOverdue`, `sumByCurrency`, аритметика на периоди) — да остане отделена от UI.
- Лексикографско сравнение на ISO дати в `isPaymentOverdue` — коректно.
- Централизирана тема (`src/theme`) и UI компоненти (`src/components/ui.tsx`).
- Последователна локализация на български.
- Валутно-разделени суми (`sumByCurrency`).
- `.gitignore` пази секрети/ключове (`.p8`, `.p12`, `.key`, `.env`).

## P1 — Да се направи ПРЕДИ Phase 4

### [ ] 1. Изравняване на версиите на зависимостите
**Проблем:** `expo-notifications` е резолвнат на `0.28.19` (версия за SDK 51), а проектът е Expo SDK 54.0.35. Натив модул с грешна версия може да чупи билда или да се държи непредвидимо.
**Действие:**
- `npx expo-doctor`
- `npx expo install expo-notifications`
- `npx expo install --check` (изравнява и останалите зависимости към SDK 54)
**След ъпгрейда** (виж задача 5) ще трябва да се обнови notification handler-ът и типът на trigger-а.
**Бележка:** `AGENTS.md` сочи към docs за `v56.0.0`, а проектът е `v54` — уеднакви референцията, за да не заблуждава Claude Code.

### [ ] 2. Миграция на ID: `int autoincrement` → UUID
**Проблем:** автоинкрементните ID-та се чупят при синхронизация между устройства (две устройства раздават един и същ ID). Отбелязано е и в STATUS.md.
**Защо сега:** данните са малко и базата може да се ресетне без болка. По-късно е тежка миграция с пренос на данни.
**Действие:** смени първичните ключове на текстов UUID (напр. генериран с `crypto.randomUUID()` или `expo-crypto`); обнови чуждите ключове и всички места, които приемат `id: number`.
**Засегнати:** `src/db/schema.ts`, `src/db/client.ts`, `src/db/utils.ts`, `src/services/*`, екраните с `[id]` параметри (`property/[id].tsx`, `tenant/[id].tsx`).

### [ ] 3. Единствен източник на истината за схемата (drizzle-kit миграции)
**Проблем:** схемата е дефинирана два пъти — в `schema.ts` (Drizzle) и ръчно като `CREATE TABLE` в `initDatabase()`. Двете ще се разминат рано или късно.
**Действие:** въведи **drizzle-kit** миграции като единствен източник; махни ръчния `CREATE TABLE` от `initDatabase()` и пускай генерирани миграции при старт.
**Засегнати:** `src/db/client.ts`, нова папка за миграции, `drizzle.config`.

### [ ] 4. Интегритет на ниво база данни
**Проблеми и действия:**
- [ ] Добави `UNIQUE(lease_id, period)` в `payments` — уникалността сега се пази само в кода, бъг в UI може да вкара дубликат.
- [ ] Добави индекси по чуждите ключове (`leases.property_id`, `leases.tenant_id`, `payments.lease_id`) — SQLite не ги индексира автоматично.
- [ ] Обмисли `ON DELETE` правила в схемата — каскадите сега са само на ниво приложение.
**Засегнати:** `src/db/schema.ts`, миграции.

### [ ] 5. Известия — корекции
- [ ] **Handler API (след задача 1):** новата версия иска `shouldShowBanner` / `shouldShowList` вместо `shouldShowAlert`. Файл: `src/services/notifications.ts`.
- [ ] **Trigger тип:** махни `as any` (`notifications.ts:69`) и ползвай типизирания trigger на новата версия.
- [ ] **Валута в тялото:** `body` показва само число — мини на `formatMoney(lease.rentAmount, lease.currency)`.
- [ ] **Тестване:** използвай **development build**, не Expo Go (от SDK 53 Expo Go е орязан откъм нотификации).
- [ ] **Документирай ограничението:** планират се само 3 месеца напред; ако приложението не се отваря >3 месеца, напомнянията спират.

## P2 — Важни подобрения (не блокиращи)

### [ ] 6. Date picker вместо текстов вход
Датите се въвеждат на ръка („ГГГГ-ММ-ДД" / „ГГГГ-ММ"). `@react-native-community/datetimepicker` маха цял клас потребителски грешки.
**Засегнати:** `property/[id].tsx` (редове ~443, ~513), формите за договор и плащане.

### [ ] 7. Edge case с края на месеца
При `paymentDay = 31` за февруари `isPaymentOverdue` сглобява несъществуваща дата „2026-02-31". Сравнението не гърми, но семантиката е разкривена. Ограничи деня спрямо реалната дължина на месеца.
**Засегнати:** `src/lib/domain.ts` (`isPaymentOverdue`).

### [ ] 8. Импорт / възстановяване на данни
Има експорт (JSON/CSV), но няма импорт/restore. Докато няма облак, изгубен/сменен телефон = изгубени всички данни. Дори само импорт на JSON покрива дупката до Phase 4.
**Засегнати:** `src/services/export.ts` (+ нов импорт), екран „Настройки".

### [ ] 9. Ефективност на справките
`generateYearlyReport` вика `generateMonthlyReport` 12 пъти, а всяко извикване чете наново всички таблици (~36 пълни сканирания за годишен отчет). Зареди данните веднъж и сметни 12-те месеца в паметта.
**Засегнати:** `src/services/reports.ts`.

### [ ] 10. CSV формат и сигурност
Един файл със секции „=== ИМОТИ ===" е четим за човек, но не се импортира чисто в Excel (различен брой колони на ред). Обмисли отделни файлове или XLSX. Когато се стигне до чужди данни — добави защита срещу CSV-инжекция (екранирай стойности, започващи с `= + - @`).
**Засегнати:** `src/services/export.ts`.

### [ ] 11. Unit тестове за `domain.ts`
Чистите функции (аритметика на периоди, `isPaymentOverdue`, форматиране на пари) са идеални за няколко теста — особено преди миграцията на ID-тата, за да се хванат регресии. Виж Expo guide за Jest.
